Error Shape

共通エラーレスポンスの見方

エラー時の JSON は、失敗の種類と対象URLをまとめて返す形にしています。

{
  "timestamp": "2026-03-27T02:00:00",
  "status": 404,
  "error": "Not Found",
  "message": "task not found",
  "path": "/tasks/9999"
}

timestamp

いつ失敗したかを表します。

status

HTTPステータスコードです。400 や 404 を数値で確認できます。

error

Bad Request や Not Found など、人が読める種別です。

message

今回の失敗理由です。例: title must not be blank

path

どのURLで起きたかを表します。ログとの突合にも便利です。

Test Matrix

今回追加したテストの全体像

「正常系だけ」ではなく、「壊れ方」まで確かめるのが今回の狙いです。 Controller は MockMvc、Service はユニットテストで確認しています。

Controller 正常系

  • タスク作成
  • 1件取得
  • 一覧取得
  • 検索
  • 更新
  • 削除

Controller 異常系

  • 空タイトルの 400
  • 壊れた JSON の 400
  • 不正な ID 型の 400
  • 存在しない ID の 404

Service テスト

  • create で timestamp が入る
  • update で title / updatedAt が変わる
  • findById の not found
  • delete の not found / success
1

Controller で外側の契約を確認

HTTP ステータス、レスポンスJSON、検索・ページングの返り方を検証します。

2

Service で重要ロジックを確認

timestamp 設定や削除ロジックの分岐など、APIの土台になる処理を押さえます。

3

Gradle でまとめて回帰確認

./gradlew test と ./gradlew build で全体が崩れていないことを見ます。

REST Upgrade

なぜ 201 Created + Location にしたのか

新規作成 API では、「作れた」だけでなく「どこに作られたか」まで伝えると、 REST API として自然で、ポートフォリオでも意図が伝わりやすくなります。

以前の返し方

200 OK で本文だけ返す

今回の返し方

201 Created と Location: /tasks/{id} を返す

POST /tasks

HTTP/1.1 201 Created
Location: http://localhost:8080/tasks/1
Content-Type: application/json

{"id":1,"title":"task"}
