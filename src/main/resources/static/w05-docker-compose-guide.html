Why app waits

DBコンテナが「起動しただけ」ではなく、「接続受付可能」になるまで待つためです。

Why volume remains

コンテナを消してもデータを残し、ポートフォリオ確認時に毎回初期化しないためです。

Why names are not fixed

container_name を外し、同名競合が起きにくい compose 標準の命名に寄せました。

Dockerfile

Dockerfile で改善したポイント

以前は 1 つの JDK イメージで build と run をまとめていましたが、 今回は役割を分けて読みやすくしました。

builder stage

gradlew と src をコピーし、bootJar を実行して jar を作ります。

runtime stage

軽い JRE イメージに jar だけを持ち込み、実行専用コンテナとして使います。

curl 追加

app コンテナ自身の healthcheck に使うために curl を入れました。

.dockerignore

.env、build/、.git などを除外し、context を軽くしています。

Compose

docker compose で改善したポイント

学習用として過度に複雑にせず、でも「実際に困るポイント」は避ける構成にしています。

healthcheck を追加

db は pg_isready、app は /actuator/health で確認します。

depends_on を強化

service_started ではなく service_healthy を使い、DB未準備で app が失敗しにくくしました。

fail-fast secrets

POSTGRES_PASSWORD、APP_SECURITY_PASSWORD、JWT_SECRET は未設定だと compose 自体が止まります。

ポート可変化

API_PORT、POSTGRES_PORT を変えれば、他プロジェクトと競合しても逃がせます。

ポイント

PostgreSQL の初期化変数を変えたのに volume が残っていると、前回の認証情報でDBが動き続けます。 その場合は docker compose down -v が必要です。

Run Modes

ローカル実行と Docker 実行をどう両立したか

「Docker では動くけどローカルでは壊れる」またはその逆、を避けるための整理も今回のテーマでした。

Docker 実行

  • DB_HOST=db を compose 側で固定
  • コンテナ内ネットワークで db:5432 に接続
  • docker compose up --build が入口

ローカル実行

  • .env の初期値は DB_HOST=localhost
  • Spring Boot が .env を自動読込
  • ./gradlew bootRun が入口
Docker向け上書き

Compose がアプリコンテナにだけ DB_HOST=db を渡します。

↔
ローカル向け既定値

.env と properties のデフォルトで localhost 接続になります。
