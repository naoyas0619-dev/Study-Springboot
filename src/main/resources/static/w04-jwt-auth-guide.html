Public Endpoints

  • POST /auth/login
  • POST /auth/refresh
  • GET /swagger-ui/**
  • GET /actuator/health

Protected Endpoints

  • /tasks/** は ROLE_USER 以上
  • /admin/users/** は ROLE_ADMIN のみ
  • POST /auth/logout は認証必須

Error Responses

  • 未認証: 401 Unauthorized
  • 権限不足: 403 Forbidden
  • rate limit: 429 Too Many Requests

Token Map

access token と refresh token の役割

2種類のtokenを分けることで、使いやすさと安全性のバランスを取っています。

access token

  • 短い有効期限で保護APIにアクセスするためのtoken
  • username / role / tokenVersion を持つ
  • logout やユーザー無効化で使えなくなる

refresh token

  • access token を再発行するためのtoken
  • DBには生値ではなくハッシュで保存
  • 1回使うとローテーションされ、古いtokenは再利用不可
POST /auth/login
{
  "username": "appuser",
  "password": "change-this-password-123"
}
{
  "accessToken": "...",
  "refreshToken": "...",
  "tokenType": "Bearer",
  "accessExpiresInSeconds": 900,
  "refreshExpiresInSeconds": 604800,
  "role": "ADMIN"
}

Hardening

今回入れたセキュリティ強化

ただJWTを返すだけではなく、「継続運用で困りやすいところ」まで埋めたのが今回のポイントです。

DBユーザー管理

app_users に username / password_hash / role / enabled / token_version を持たせています。

logoutによる revoke

tokenVersion を進めて、発行済み access token を使えなくします。

refresh token ローテーション

再発行時に古い refresh token を無効化し、使い回しを防ぎます。

DB永続化された rate limit

auth_rate_limits に失敗回数を持たせ、再起動後や複数インスタンスでも維持します。

1

失敗回数をDBに保存

in-memoryではなくDBに持つことで、アプリの寿命に依存しない制限にしました。

2

ユーザー無効化の即時反映

JWTに入っている情報だけを信じず、現在のDBユーザー状態も見て拒否します。

3

監査ログを残す

login success / failure、rate limit、logout、invalid token をログに出します。

Admin API

なぜ ADMIN 向けユーザー管理APIを足したのか

bootstrap user だけでは「最初の1人」は作れても、 その後の運用の見せ場が弱くなります。そこで、管理者が利用者を追加・更新できるAPIを入れました。

List

GET /admin/users

登録済みユーザーの一覧を確認できます。

Create

POST /admin/users

新しい USER / ADMIN を作成できます。

Update

PATCH /admin/users/{id}

role / enabled / password を更新できます。

Revoke

更新時はtokenも失効

無効化やパスワード変更時は tokenVersion を進めて refresh token も削除します。

PATCH /admin/users/{id}
{
  "enabled": false
}

Test Matrix

今回追加・拡張したテスト

W04では正常系だけでなく、認証まわりの壊れ方や運用上の更新反映もテストしています。

認証基本

  • ログイン成功で token pair を返す
  • JWTなしで 401
  • 壊れたJWTで 401

refresh / revoke

  • refreshで新しい token pair を返す
  • 使い終わった refresh token の再利用拒否
  • logout 後の access / refresh token 無効化

運用変更の反映

  • 無効化ユーザーの access token 拒否
  • 無効化ユーザーの refresh 拒否
  • 管理者更新で token revoke

管理者API

  • ADMIN がユーザー作成できる
  • 重複usernameは 409
  • USER は管理APIに入れず 403

Commands

確認に使うコマンド

W04の追加内容を自分で触る時に役立つコマンドです。

ログイン

curl -X POST http://localhost:8080/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"appuser","password":"change-this-password-123"}'

Task API利用

curl http://localhost:8080/tasks \
  -H "Authorization: Bearer ${ACCESS_TOKEN}"

refresh

curl -X POST http://localhost:8080/auth/refresh \
  -H "Content-Type: application/json" \
  -d '{"refreshToken":"..."}'

管理者でユーザー作成

curl -X POST http://localhost:8080/admin/users \
  -H "Authorization: Bearer ${ACCESS_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"username":"member-user","password":"member-password","role":"USER"}'
