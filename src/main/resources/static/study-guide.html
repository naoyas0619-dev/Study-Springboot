Overview

プロジェクト全体の処理の流れ

このアプリは、外から来たHTTPリクエストを Controller が受け取り、 Service で処理し、Repository を通して Database に保存します。

1. クライアント

ブラウザ / curl / Swagger UI

→
2. Controller

URLを受け取り、どの処理を呼ぶか決める

→
3. Service

作成・更新・検索などの業務ロジックを担当

→
4. Repository

DBへ保存・検索する窓口

→
5. Database

PostgreSQL の tasks テーブル

入口

TaskController が /tasks へのアクセスを受けます。

処理

TaskService が日時設定や存在確認などの処理を行います。

保存

TaskRepository が JPA を通してSQL発行を担当します。

Architecture

Controller / Service / Repository の関係図

このプロジェクトでは、画面やAPIの入口と、業務処理と、DBアクセスを分けています。 それぞれの責任を分けることで、コードが読みやすくなり、修正もしやすくなります。

Client

Browser / curl / Swagger UI

HTTPリクエストを送る利用者側です。

↓
Controller Layer

TaskController

URLとHTTPメソッドを受け取り、どの処理を呼ぶか決めます。

POST /tasks GET /tasks PUT /tasks/{id}
↓
Service Layer

TaskService

日時の設定、存在確認、更新処理などの業務ロジックを担当します。

create() findById() update() delete()
↓
Repository Layer

TaskRepository

JPAを通して、保存・検索・削除などのDB操作を行います。

save() findAll() findByTitleContaining()
↓
Database Layer

Task Entity ↔ PostgreSQL

Task.java が tasks テーブルに対応し、データの形を表します。

Controller と DTO の関係

TaskController は、受け取る時に TaskRequest、 返す時に TaskResponse を使います。

Service と Entity の関係

TaskService は、内部処理では Task Entity を使ってDB保存用のデータを扱います。

例外処理の流れ

NotFoundException やバリデーションエラーは GlobalExceptionHandler が受けて、HTTPエラーに変換します。

API Explorer

各APIを押して、対応する処理を見る

下のAPIボタンを押すと、そのAPIがどのControllerメソッドを呼び、 どのService処理につながっているかがハイライトされます。

Create Flow

POST /tasks

新しいタスクを作成するAPIです。

Controller TaskController#create

受け取ったJSONを TaskRequest に変換し、Serviceへ渡します。

→
Service TaskService#create

createdAt と updatedAt を設定して保存準備をします。

→
Repository TaskRepository#save

JPAがINSERT相当の保存処理を行います。

Read Flow

GET /tasks

タスク一覧をページ付きで取得するAPIです。タイトル検索にも対応しています。

Controller TaskController#getAll

ページ情報と検索条件を受け取り、どのServiceメソッドを呼ぶか分岐します。

→
Service TaskService#findAll / search

全件取得かタイトル検索かを実行し、結果を TaskResponse に変換します。

→
Repository findAll / findByTitleContaining

ページング付きでDBからレコードを取得します。

Read Flow

GET /tasks/{id}

指定したIDのタスクを1件取得するAPIです。

Controller TaskController#getById

パス変数のIDを受け取り、Serviceへそのまま渡します。

→
Service TaskService#findById

存在しない場合は NotFoundException を投げます。

→
Repository findById

IDで検索し、見つかった1件を返します。

Update Flow

PUT /tasks/{id}

既存タスクのタイトルを更新するAPIです。

Controller TaskController#update

更新対象のIDと新しいタイトルを受け取ります。

→
Service TaskService#update

対象を探してタイトルを書き換え、updatedAt を更新します。

→
Repository save

更新済みEntityを再保存して、UPDATE相当の処理を行います。

Delete Flow

DELETE /tasks/{id}

指定IDのタスクを削除するAPIです。

Controller TaskController#delete

IDを受け取り、削除処理をServiceへ依頼します。

→
Service TaskService#delete

存在確認をしてから削除します。見つからなければ例外になります。

→
Repository existsById / deleteById

存在確認と削除をDBに対して行います。

Request Flow

HTTPリクエストが来てからDBに保存されるまで

ここでは POST /tasks でタスクを作成するときの流れを、 1ステップずつ分解しています。

送られてくる例 
POST /tasks
Content-Type: application/json

{
  "title": "Spring Bootを理解する"
}
01

クライアントがリクエストを送る

ブラウザ、curl、Swagger UI などから POST /tasks が送られます。

02

Controller が受け取る

TaskController#create がURLとJSONを受け取り、次の処理へ渡します。

03

TaskRequest に変換される

JSONの title が TaskRequest に入ります。空文字なら @NotBlank でエラーになります。

04

Service が業務処理を行う

TaskService#create が createdAt と updatedAt をセットします。

05

Repository が保存する

repo.save(task) を実行すると、Spring Data JPA がDB保存処理を組み立てます。

06

PostgreSQL にレコードが入る

tasks テーブルに新しい1件が保存され、IDが自動採番されます。

07

レスポンス用の形に変換する

TaskResponse に変換して、クライアントに返す項目だけを絞ります。

08

JSONレスポンスを返す

最終的に {"id": 1, "title": "..."} のようなJSONが返ります。

DB Change

Task作成時のJSONレスポンス例とDBテーブル変化

ここでは POST /tasks を実行したときに、 「返ってくるJSON」と「DBの中身がどう変わるか」を並べて見られます。

Response Example

クライアントに返るJSON

{
  "id": 1,
  "title": "Spring Bootを理解する"
}

このAPIでは、レスポンスには id と title を返しています。 createdAt や updatedAt は内部管理用なので返していません。

Table Change

tasks テーブルの変化

保存前

id title created_at
まだレコードなし
→

保存後

id title created_at
1 Spring Bootを理解する 2026-03-25 21:00...
受信したJSON

{"title":"Spring Bootを理解する"}

→
Task Entity

title と日時がセットされる

→
INSERT

JPAがDB保存処理を実行する

→
TaskResponse

返却用に id と title へ整理

File Map

どのファイルが何をしているのか

まずはこの順番で読むと、プロジェクトの全体像をつかみやすいです。

アプリの入口

  • TaskApiApplication.java: Spring Bootアプリを起動するメインクラス
  • StudyGuideController.java: この学習ページを開くためのルート
  • HelloController.java: 動作確認用のシンプルなAPI

APIの本体

  • TaskController.java: HTTPリクエストの受付窓口
  • TaskService.java: 業務処理の中心
  • TaskRepository.java: DBアクセス担当
  • Task.java: DBの tasks テーブルに対応するEntity

受け渡し用のクラス

  • TaskRequest.java: リクエストで受け取るデータ
  • TaskResponse.java: レスポンスで返すデータ
  • ErrorResponse.java: エラー時に返すJSONの形

エラー処理

  • NotFoundException.java: データが見つからない時の独自例外
  • GlobalExceptionHandler.java: 例外をHTTPエラーに変換するクラス

設定ファイル

  • application.properties: DB接続やJPAの設定
  • build.gradle: 依存ライブラリやビルド設定
  • docker-compose.yml: APIとPostgreSQLをまとめて起動する設定
  • Dockerfile: APIのDockerイメージを作る手順
